# Regenerating the README demos

The README hero is a static screenshot; the two demo GIFs are short clips. All
three are generated from the **real `copilotline` CLI output** (the GIFs via
[charmbracelet VHS](https://github.com/charmbracelet/vhs)), so they can never
silently drift from shipped behavior:

- `docs/screenshot.png` — a static, rounded dark card of the statusline ribbon
  (the README hero). Regenerated by `bash docs/fixtures/make-screenshot.sh`; see
  [The static hero screenshot](#the-static-hero-screenshot) below.
- `docs/demo-statusline.gif` — the full statusline ribbon (the real
  `dist/cli.js render` against an offline, public-safe usage-cache fixture).
- `docs/demo-cli.gif` — a `copilotline doctor` report that is green for
  Environment, Configuration, and Account (the real `dist/cli.js doctor`) in a
  fully isolated, anonymized environment.

The committed assets are the source of truth between regenerations. Regeneration
is a manual maintainer step; CI does not render the demos (VHS needs a TTY plus
`ffmpeg`).

## What appears on screen

Each GIF shows the installed experience. After the `Show`, the only text typed
on screen is a single command (there is **no** synthetic `copilotline install`
step on screen — the install runs off screen as part of setup, see below):

- `docs/demo-statusline.gif`:
  - `echo "$PAYLOAD" | copilotline render` — emits the full statusline ribbon
    (`$PAYLOAD` is the public-safe status payload, loaded off screen).
- `docs/demo-cli.gif`:
  - `copilotline doctor` — prints the diagnostics report.

All of the demo plumbing is built OFF screen, before VHS runs, by the wrapper
`docs/fixtures/render-demos.sh` (which calls `docs/fixtures/seed-demo-shell.sh`
and, under it, `docs/fixtures/seed-demo-harness.mjs`). Each tape then types only
one short fixed line off screen — `source $CL_DEMO/env.sh` — plus the single
visible command.

### Why setup runs off screen (do not type it in the tape)

VHS drives an interactive bash through a PTY and types keystrokes **faster than
the shell consumes a command's output**. If the multi-step setup is typed inside
the tape (whether split across many `Type … Enter` lines or collapsed into one
long line), the next command's keys intermittently interleave with the previous
command's still-streaming output and silently corrupt the line. When that
corrupts the payload assignment, `echo "$PAYLOAD"` emits an empty line, so
`copilotline render` reads empty stdin, **falls back to the model "Copilot", and
reads the REAL host Copilot account** — a PII leak straight into the GIF.

The fix is structural, not a sleep: `docs/fixtures/render-demos.sh` performs the
entire setup in a normal (non-PTY) shell, exports the isolated demo root as
`$CL_DEMO`, and invokes `vhs`. VHS inherits the parent environment, so the tape
sees `$CL_DEMO` and only has to `source $CL_DEMO/env.sh` (one short, fixed,
non-wrapping line) to pull in `cd`, the isolated env, and the public-safe
`$PAYLOAD`. With nothing fragile left to type, the renders are deterministic and
the payload is guaranteed populated before the visible `render`.

## The anonymized, offline, PII-free harness

Three committed helpers layer together:

- **`docs/fixtures/render-demos.sh`** — the wrapper you actually run. For each
  demo it builds the harness off screen, exports `$CL_DEMO`, runs `vhs`, then
  removes the `/tmp` demo root.
- **`docs/fixtures/seed-demo-shell.sh`** — the off-screen setup. It runs the node
  harness below, creates the throwaway git repo, runs `copilotline install`
  against the isolated env, and writes `$CL_DEMO/env.sh` (the `cd` + isolated env
  exports + the public-safe `$PAYLOAD`) for the tape to `source`.
- **`docs/fixtures/seed-demo-harness.mjs`** — the file writer described next.

`docs/fixtures/seed-demo-harness.mjs` builds a throwaway demo root under
`/tmp/copilotline-demo-*` so the demo paths are identical on every machine (no
`$HOME`, no per-user temp hash). It writes:

- **`<demo>/cli.js`** — a copy of the built `dist/cli.js`. The demo runs the CLI
  through this copy, so the `statusLine.command` that `copilotline install`
  records in `settings.json` (derived from `import.meta.url`) is the anonymized
  `/tmp/.../cli.js` path, never the real repository path. `copilotline doctor`
  then reports that command as wired and the configured executable as found.
- **`<demo>/bin/copilotline`** — a shim on `PATH` so
  `isCommandAvailable("copilotline")` is green and the typed command reads
  exactly like the installed binary. For the `install` subcommand the shim
  appends `--no-account`, so the off-screen install runs **non-interactively**:
  without that flag `copilotline install` would open the interactive account
  wizard ("Choose quota account") and block.
- **`<demo>/bin/gh`** — a stub that exits non-zero, so `accountFromGitHubCli`
  returns null. No host GitHub account can leak into the demo and there is no
  slow `gh auth status` spawn.
- **`<demo>/bin/copilot`** — the public-safe version stub from
  `docs/fixtures/copilot-shim.sh`, so doctor's `copilot command available` line
  is deterministic and no host Copilot version leaks into the GIF.
- **`<demo>/copilot/config.json`** — a fabricated `lastLoggedInUser` of the
  well-known demo login **`octocat`** on `github.com`, so account detection
  resolves a public-safe login with no PII. This is what turns the doctor
  Account section and the ribbon credits segment green without a real account.
- **`<demo>/cache/github.com-octocat.usage-cache.json`** — an offline credits
  snapshot keyed to `octocat`, mirroring `docs/fixtures/usage-cache.json` and
  stamped with a fresh `fetchedAt` (within TTL). The credits ribbon and the
  doctor billing-unit line render from this fixture with **no network and no
  token**. All values are fabricated and public-safe.
- **`<demo>/payload.json`** — the public-safe status payload (`gpt-5.5` /
  `xhigh`, ~47% context, cwd in a throwaway `copilotline` git repo on branch
  `main`, and a `started_at` ~2h27m ago so the timer segment is deterministic).

`seed-demo-shell.sh` prepends `<demo>/bin` to `PATH` (so the shim and stubs win
while coreutils stay available) and exports an isolated `COPILOT_HOME`,
`COPILOTLINE_CACHE_DIR`, and a nonexistent `COPILOTLINE_VSCODE_STATE_DB` so VS
Code account detection is skipped too; it records all of that, plus the `cd` and
`$PAYLOAD`, into `<demo>/env.sh` for the tape to `source`. Nothing touches the
network, a GitHub token, or a real account.

### Why doctor is green (and what stays an honest WARN)

After the off-screen `copilotline install`, `copilotline doctor` reports
**12 pass, 1 warn, 0 fail**:

- **Environment** — `copilotline`, `copilot`, and `git` on `PATH`, plus Node:
  all green via the shim and stubs.
- **Configuration** — settings file found, `statusLine.command` wired to the
  anonymized `/tmp/.../cli.js`, and `footer.showCustom` enabled: all green,
  because the off-screen `copilotline install` wrote them this run.
- **Account** — `octocat` detected, quota mode `auto`, AI-credits billing: all
  green from the fabricated config + offline cache.
- The **single honest WARN** is the quota-token check
  (`No quota token available for octocat`). Validating a token requires a live
  GitHub API call, so the demo never fabricates one — that one check stays an
  honest warn rather than faking a credential.

## Install the tooling

```bash
# macOS
brew install vhs ffmpeg

# Linux: see https://github.com/charmbracelet/vhs#installation
```

`vhs` also needs `ttyd` on the `PATH`; `brew install vhs` pulls it in.
ImageMagick 7 (`magick`) is used to verify the canvas fits the content.

## Regenerate the GIFs

Run from the repository root. Use the wrapper — **not** bare `vhs` — because the
wrapper builds the harness off screen (see [Why setup runs off
screen](#why-setup-runs-off-screen-do-not-type-it-in-the-tape)). It builds the
bundle, then renders both tapes:

```bash
bash docs/fixtures/render-demos.sh
```

Each tape writes its GIF to `docs/demo-*.gif` (the `Output` path is resolved
relative to the directory you invoke `vhs` from — i.e. the repo root — not the
tape's own directory). The new GIFs replace the committed ones in place; their
filenames are stable so the README `raw.githubusercontent.com` image URLs keep
working.

> Running `vhs docs/demo-statusline.tape` directly will fail: the tape expects
> `$CL_DEMO` to be set and `$CL_DEMO/env.sh` to exist, both of which the wrapper
> provides.

## The static hero screenshot

`docs/screenshot.png` is the README hero: a static, rounded dark card of the
statusline ribbon. Regenerate it with:

```bash
bash docs/fixtures/make-screenshot.sh
```

It uses the **same PII-free `octocat` harness** as the GIFs: it builds the
isolated, fabricated-`octocat` environment via `docs/fixtures/seed-demo-
harness.mjs` and renders the **real** `copilotline render` output OUTSIDE VHS
(so it can never race or fall back to the host account), guards that the ribbon
contains `octocat` and no real account/path, then has VHS draw only that static
ANSI (VHS renders the emoji/glyphs faithfully) before ImageMagick trims, pads,
and rounds the corners into the card. It writes `docs/screenshot.png` in place;
the filename is stable so the README hero URL keeps working.

### Pixel-perfect canvas

Both tapes are sized so the GIF canvas fits the content exactly — no larger and
no smaller — with a small uniform padding (`Set Padding 24`). The current
converged sizes are `1352x128` (statusline) and `1844x612` (doctor). To
re-converge after an output change, regenerate via the wrapper, extract the
final held frame, and trim:

```bash
bash docs/fixtures/render-demos.sh
magick "docs/demo-<name>.gif" -coalesce -delete 0--2 /tmp/lastframe.png
# robust against the GIF palette's edge dithering:
magick /tmp/lastframe.png -bordercolor "srgb(28,28,44)" -border 1 -fuzz 8% \
  -trim -format "%wx%h+%X+%Y\n" info:
```

Set `Set Width = contentW + 2*Padding` and
`Set Height = contentH + 2*Padding` (add a few px of width safety so the longest
line — the ribbon, or doctor's synthetic-render preview — is not wrapped, and a
few px of height so the vertical border matches). VHS adds a small fixed
vertical margin on top of `Set Padding`, so the trimmed border is uniform per
axis (horizontal ≈ Padding, vertical ≈ Padding + a few px) with nothing clipped.
Keep the canvas tall enough that the held frame shows the typed command line and
its output together without scrolling the command off the top.

## How the demos stay PII-free

The tapes, the screenshot, and the harness use **public-safe sample values
only** and never touch the network, a GitHub token, or a real account.
Everything is built by `docs/fixtures/seed-demo-harness.mjs` described above; the
fabricated account is the public demo login `octocat`, the credits come from
`docs/fixtures/usage-cache.json`, and all paths are fixed
`/tmp/copilotline-demo-*` temp paths.

The setup runs OFF screen (see above), which also closes the one PII hole that
matters here: a corrupted, empty payload makes `copilotline render` fall back to
the model "Copilot" and read the REAL host Copilot account. After regenerating,
verify with `gitleaks detect --no-git --source docs` and confirm the rendered
frames show only `octocat` (no real login, no `/Users/...` path, no "Copilot "
fallback model).

**Rule:** never paste real tokens, raw Copilot captures, usernames, or private
repository paths into a `.tape` script, a fixture, or a rendered asset. Use
anonymous, fixed temp paths and the public-safe `octocat` login.

## When to regenerate

Regenerate whenever the **visible output changes**:

- the statusline ribbon changes (a segment is added/removed/reformatted) →
  regenerate `docs/demo-statusline.gif` and `docs/screenshot.png`.
- the `copilotline doctor` report changes (a section, line, or label) →
  regenerate `docs/demo-cli.gif`.

There is no need to regenerate for changes that do not alter rendered output.
